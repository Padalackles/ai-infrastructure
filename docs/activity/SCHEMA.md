## Event Schema (v1)

```json
{
  "version": 1,
  "id": "evt_01JX2K4N8P3Q5R7S9T",
  "timestamp": "2026-06-19T09:00:00.000Z",
  "source": "android",
  "collector": "macrodroid",
  "device": "pixel-8-pro",
  "type": "device.awake",
  "payload": {},
  "raw": {}
}
```

### Field Reference

| Field | Type | Required | Description |
|---|---|---|---|
| `version` | `integer` | âœ… | Schema version.  Bump when fields are added/removed. |
| `id` | `string` | âœ… | Globally unique event ID.  Generated by the Activity Gateway on ingest.  Format: `evt_<ULID>` |
| `timestamp` | `string` (ISO 8601) | âœ… | When the event **occurred** on the device.  Millisecond precision. |
| `source` | `string` | âœ… | Originating platform.  One of: `android`, `ios`, `desktop`, `web`, `iot`, `service`. |
| `collector` | `string` | âœ… | Software that captured the event.  Examples: `macrodroid`, `tasker`, `shortcuts`, `home-assistant`, `scheduler`. |
| `device` | `string` | âœ… | Human-readable device identifier.  Examples: `pixel-8-pro`, `iphone-15`, `thinkpad-x1`. |
| `type` | `string` | âœ… | Hierarchical event type (see [Naming Convention](#naming-convention)). |
| `payload` | `object` | âœ… | Normalized, typed event data.  Schema varies by `type`. |
| `raw` | `object` | âœ… | Original, unmodified event as received from the collector.  Always preserved for debugging. |

### Design Rules

1. **Every field is required.**  No optional fields.  Empty values use their zero-value: `""` for strings, `{}` for objects, `0` for numbers.
2. **`payload` is typed per `type`.**  Each event type defines its own payload sub-schema.  Unknown fields are silently dropped by the Normalizer.
3. **`raw` is never normalized.**  It carries the collector-specific format exactly as received.  This guarantees we can re-process historical events if the Normalizer logic changes.
4. **`id` is generated server-side.**  Collectors never supply IDs â€” the Gateway assigns a ULID on ingest, guaranteeing global uniqueness and time-sortability.

## Naming Convention

Event types use **hierarchical dot-notation**: `<domain>.<subdomain>.<action>`.

```
device.awake
device.sleep

battery.low
battery.full
battery.charging.started
battery.charging.stopped

app.opened
app.closed

network.connected
network.disconnected

location.changed

notification.received
notification.dismissed

call.started
call.ended
call.missed

screen.on
screen.off
screen.unlocked
```

### Rules

1. **Lowercase only.**  `device.awake`, never `Device.Awake`.
2. **Domain first.**  The leftmost segment is the broad category.
3. **Action last.**  The rightmost segment is the specific event.
4. **Past tense for actions.**  `app.opened`, not `app.open`.
5. **Max 4 segments.**  `battery.charging.started` is valid; `a.b.c.d.e` is rejected.
6. **New types are additive.**  Adding `bluetooth.connected` requires no changes to existing handlers.

### Reserved Domains

| Domain | Scope |
|---|---|
| `device` | Device-level events (awake, sleep, boot, shutdown) |
| `battery` | Power state changes |
| `app` | Application lifecycle |
| `network` | Connectivity changes |
| `location` | Geolocation updates |
| `notification` | Incoming notification events |
| `call` | Telephony events |
| `screen` | Display state changes |
| `sensor` | Physical sensor readings |
| `bluetooth` | Bluetooth device events |
| `wifi` | Wi-Fi network events |
| `schedule` | Internally generated scheduler events |

## Design Principles

### 1. Source Agnostic

The schema makes no assumptions about:
- Operating system (Android, iOS, desktop)
- Collector software (MacroDroid, Tasker, Home Assistant)
- Transport mechanism (HTTP webhook, WebSocket, MQTT)

Adding a new source requires zero schema changes.

### 2. Normalize Late

Raw events are never discarded.  The Normalizer produces `payload` from
`raw`, but `raw` is always preserved.  This lets us:
- Re-process events if the Normalizer logic changes
- Debug collector-specific quirks
- Audit the normalization pipeline

### 3. Schema-Versioned

The `version` field is an integer that increments when:
- A required field is added
- A field type changes
- A field is removed

Adding an optional field or a new event type does **not** require a version
bump.  Consumers that encounter an unknown `version` must reject the event.

### 4. Typed Payload

Every event `type` maps to a specific `payload` sub-schema.  This lets
Decision Scripts switch on `type` and safely access typed fields:

```
if event.type == "battery.low":
    level = event.payload.level   # guaranteed to exist
```

### 5. Extensible by Addition

New event types are added by:
1. Defining a payload sub-schema
2. Adding the type to the reserved domain list
3. Updating the Normalizer mapping

No existing handlers, routes, or schemas change.

## Future Extension Strategy

### Adding a New Source (e.g., iOS)

1. Collector sends events to the Gateway in any format.
2. Gateway wraps the raw event with `source: "ios"`.
3. Normalizer maps iOS-specific fields to the unified schema.
4. Zero schema changes required.

### Adding a New Collector (e.g., Tasker)

Same as above â€” set `collector: "tasker"` and map fields in the Normalizer.

### Adding a New Event Type (e.g., `health.step_count`)

1. Define the payload sub-schema in this document.
2. Add `health` to the reserved domains table.
3. Update the Normalizer mapping.
4. Decision Scripts may now react to `health.step_count`.

No version bump required â€” new types are additive.

### Schema v2 (future)

If a breaking change is needed:
1. Bump `version` to `2`.
2. Document the migration path from v1 â†’ v2.
3. The Gateway emits v2 events; v1 consumers reject them safely.
