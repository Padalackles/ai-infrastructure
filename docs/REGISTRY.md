# MCP Registry

## Purpose

The MCP Registry is the Hub's source of truth for what MCP services exist, which are enabled, and how they are configured.

This is a **design document** — no implementation exists yet. The Registry will be built in Phase 3.

---

## Design

### Directory Structure

```
registry/
├── manifests/
│   ├── ombre.yaml
│   ├── ntfy.yaml
│   ├── filesystem.yaml
│   ├── github.yaml
│   └── browser.yaml
├── installed.yaml
└── enabled.yaml
```

### Manifest Format (`manifests/<service>.yaml`)

```yaml
# ombre.yaml
name: ombre
version: "0.1.0"
class: OmbreServer
description: Long-term AI memory
capabilities:
  - tools
  - resources
endpoint: http://localhost:3000
enabled: true
```

### Installed Registry (`installed.yaml`)

Auto-generated by Discovery at startup. Lists every discovered server.

```yaml
installed:
  - name: ombre
    version: "0.1.0"
    path: mcp_servers/ombre/
    discovered_at: "2026-06-17T12:00:00Z"
  - name: ntfy
    version: "0.1.0"
    path: mcp_servers/ntfy/
    discovered_at: "2026-06-17T12:00:00Z"
```

### Enabled Registry (`enabled.yaml`)

User-managed. Controls which discovered servers are active.

```yaml
enabled:
  - ombre
  - ntfy
# github is installed but disabled
```

---

## Lifecycle

```
Discover (scan mcp_servers/)
    ↓
Validate (manifest.yaml)
    ↓
Register (ServerManager)
    ↓
Enable/Disable (enabled.yaml)
    ↓
Initialize → Start → Running
    ↓
Stop → Drain → Unregister
```

Each MCP server follows this lifecycle. The Registry tracks state transitions.

---

## Hub Responsibilities

- **Discover** MCPs at startup
- **Register** validated servers
- **Enable/Disable** per configuration
- **Manage Lifecycle** (init → start → stop)
- **Report Status** via /health and /status

The Registry does NOT:
- Implement business logic
- Store user data
- Make routing decisions
